# WBS強化 + JIRA CSVエクスポート 設計書

**作成日:** 2026-04-14  
**対象ファイル:** `gantt-generator.html`（単一ファイル構成）  
**アプローチ:** Approach B — WBS体験強化 + JIRAエクスポート

---

## 背景・目的

このツールは「WBSを計画してガントチャートにする」フェーズを担う。その後のワークフローは：

1. 本ツールでタスク分解・担当アサイン・スケジュール計算
2. JIRA チケットを作成して担当者に割り当て
3. JIRA 上で進捗管理

現状は JSON / HTML / PNG エクスポートのみ。JIRA チケット作成が手作業になっているため、CSV エクスポートで一括インポートできるようにする。あわせて WBS 番号の自動付番でガントチャートの可読性と JIRA チケット名の一貫性を向上させる。

---

## JIRA チケット階層マッピング（デフォルト）

```
Epic（既存・プロジェクト設定で指定）
  └─ Task（アイテム単位）
       └─ Sub-task（フェーズ単位）
```

- Epic は1プロジェクトにつき1つ。プロジェクト設定タブで Epic キーを入力する。
- 将来的にアイテムごとに Epic を変えたいニーズが出た場合は拡張ポイントとして想定済み（今回はスコープ外）。

---

## セクション1: データモデルの変更

### プロジェクト設定（`C`）に追加

```jsonc
{
  "epicKey": "PROJ-1",   // JIRA Epic キー。空欄可（空欄時は Epic Link 列が空）
  ...
}
```

`DEFAULT_CONFIG` にも `epicKey: ""` を追加する。

### 担当者（`C.people[]`）に追加

```jsonc
{
  "name": "Alice",
  "jiraUser": "alice@example.com",  // JIRA ユーザー名またはメールアドレス
  ...
}
```

- `jiraUser` が空欄の場合は `name` をそのまま Assignee として使用する。
- JIRA Cloud はメールアドレス、JIRA Server はユーザー名が一般的。ツール側では区別しない。

### WBS番号

- config には保存しない。レンダリング・CSV 生成時にアイテムの配列インデックスから動的に計算する。
- アイテム: 1始まりの整数（`1`, `2`, `3`…）
- フェーズ: `アイテム番号.フェーズ番号`（`1.1`, `1.2`…）

---

## セクション2: WBS番号付番

### 付番ルール

| 要素 | 番号形式 | 例 |
|---|---|---|
| アイテム | `N` | `1`, `2`, `3` |
| フェーズ | `N.M` | `1.1`, `1.2` |

### 影響箇所

- `runSchedule()` の戻り値の各タスクに `wbsNo`（文字列）を追加する。
- **ガントチャートの行ラベル**: `1.2 設計開発` の形式（アイテム行は `1 コア機能A`）
- **サマリーテーブルの「アイテム / 工程」列**: 番号を先頭に埋め込む（列追加なし）
- アイテムの追加・削除・並び替えで自動再付番される。

## セクション3: JIRA CSV フォーマット

### ファイル名

`{プロジェクト名}.csv`（JSON 保存と同じ命名規則）

### 列構成

| 列名 | Task（アイテム） | Sub-task（フェーズ） |
|---|---|---|
| `Issue Type` | `Task` | `Sub-task` |
| `Summary` | `1 コア機能A` | `1.1 要件定義 — コア機能A` |
| `Epic Link` | `PROJ-1`（epicKey） | 空 |
| `Parent` | 空 | `1 コア機能A`（親Taskと一致） |
| `Assignee` | 最初のフェーズの担当者の jiraUser | 当該フェーズの担当者の jiraUser |
| `Start Date` | スケジュール計算の開始日（YYYY-MM-DD） | 同左 |
| `Due Date` | スケジュール計算の終了日（YYYY-MM-DD） | 同左 |
| `Story Points` | アイテム全フェーズの合計工数（日数） | フェーズ工数（日数） |
| `Description` | `カテゴリ: XX\nメモ: YY` | `担当チーム: XX\n稼働日数: N日` |
| `Labels` | アイテムのカテゴリ | フェーズタイプのチーム名 |

### 特殊ケース

- **`background: true`（並行進行タスク）**: Assignee を空にし、Description に `（バックグラウンドタスク）` と付記する。
- **`requireAll: true`（複数人同時着手）**: Assignee に担当者全員をカンマ区切りで列挙する（JIRA のインポート仕様に依存するため、ユーザーが手動調整する前提）。
- **`epicKey` 未設定**: CSV の Epic Link 列を空にしてダウンロードを続行する。ボタン押下時に `alert` で「Epic キーが未設定です。このまま続行しますか？」と確認する。

### 行の並び順

Task → その Task に属する Sub-task の順でインターリーブして並べる（JIRA のインポートが Parent を Summary で解決するため）。

```
Task, 1 コア機能A
Sub-task, 1.1 要件定義 — コア機能A
Sub-task, 1.2 テストケース作成 — コア機能A
Sub-task, 1.3 設計開発 — コア機能A
Task, 2 コア機能B
Sub-task, 2.1 要件定義 — コア機能B
...
```

## セクション4: UI変更

### ① プロジェクト設定タブ

「リリース日」入力欄の直下に `Epic キー` フィールドを追加する。

```
Epic キー  [________________]  例: PROJ-1
```

### ② 担当者タブ

各担当者の編集フォームに `JIRA ユーザー名` フィールドを追加する。

```
JIRA ユーザー名  [________________]  例: alice@example.com（省略可）
```

### ③ フッターボタン列

```
[HTML 出力] [PNG 出力] [JIRA CSV] [JSON 保存] [JSON 読込]
```

- 「JIRA CSV」ボタンを `btn-png` と `btn-json-save` の間に追加する。
- スタイルは既存の `btn-secondary` を使用。
- クリック時の処理:
  1. `epicKey` が未設定なら `confirm()` で続行確認
  2. スケジュールを計算（`runSchedule()` の結果を使用）
  3. CSV 文字列を生成して BOM 付き UTF-8 でダウンロード（Excel での文字化け防止）
